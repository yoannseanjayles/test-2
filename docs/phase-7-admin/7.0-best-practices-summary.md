# Phase 7.0 — Best Practices Summary : Administration (back-office)

> **Statut** : 🟡 En attente de validation HITL
> **Équipe** : Architecte Back-end, UX/UI Designer, Product Owner, DPO/Juriste
> **S'appuie sur** : rôles D-017, statuts D-016, modèle de données Phase 6, config unique D-039, curation D-008/D-025.

## 1. Approche (proposition à valider)

**Back-office interne dans le monolithe** (`/admin`, non indexé, accès par rôle) plutôt qu'un CMS externe : mêmes tables Postgres, mêmes composants UI (design system 4.1), pas de synchronisation à maintenir. Auth = Better Auth existant + champ `role` sur l'utilisateur (Admin, Ops, Catalogue, Éditorial — D-017).

## 2. Périmètre fonctionnel (jalons 7.1)

1. **Socle admin + Catalogue** : layout `/admin` avec garde par rôle ; CRUD produits (variantes taille×couleur H14, stocks, rang « Notre sélection » H17, note de curation obligatoire D-025), catégories, revalidation ISR à la sauvegarde (< 60 s, 5.0 §4).
2. **Import AliExpress par drag & drop de pages téléchargées** *(demande utilisateur — voir §3)*.
3. **Commandes & Ops** : liste/détail, transitions de statuts D-016 (notification client à chaque étape), retours (validation, remboursement Stripe), alertes stock bas et restock à honorer (H15).
4. **Éditorial & réglages** : guides en base (migration depuis `lib/guides`, gabarit E-E-A-T D-037), réglages livraison/seuils (config unique D-039), export newsletter.

## 3. Import de produits depuis AliExpress (D-052 à valider)

**Besoin** : importer un ou plusieurs produits AliExpress **sans scraping** : l'utilisateur enregistre les pages produit depuis son navigateur (« Enregistrer sous » → HTML complet, ou `.mhtml`) puis les **glisse-dépose** (ou sélectionne) dans le back-office.

**Fonctionnement proposé** :
1. « Catalogue → Importer » : zone de drag & drop acceptant 1–N fichiers (`.html`, `.mhtml`, ou dossier « page complète » zippé) — l'upload multiple traite chaque page comme un produit.
2. Le serveur analyse chaque fichier **hors ligne** : extraction du JSON embarqué des pages AliExpress (`window.runParams` / données structurées) avec repli sur le DOM (titre, prix, variantes, images — récupérées depuis le fichier ou re-téléchargées depuis leurs URLs) ; aucune requête vers AliExpress n'est nécessaire.
3. Chaque import crée un **brouillon** (statut `draft`, jamais publié directement) : traduction/réécriture FR, prix de vente à fixer (prix fournisseur stocké à part), mapping catégorie/gabarits, sélection d'images (remplacées à terme par la DA maison, D-042), **note de curation obligatoire** avant publication (D-008 : l'import accélère la saisie, il ne court-circuite pas la sélection).
4. Import en lot : traitement séquentiel avec rapport par fichier (réussi / à compléter / échec + raison), rien n'est perdu en cas d'échec partiel.

**Avantages de l'upload vs URLs** : zéro anti-bot, fonctionne même si AliExpress change ses protections, l'utilisateur voit exactement la version de page importée. **Vigilance** : structure des pages variable selon les versions AliExpress (parseur défensif + fallback saisie manuelle) ; contenus et images fournisseur réécrits/remplacés avant publication (CGU, droits, SEO).

## 4. Sécurité & UX admin

Garde serveur sur chaque action (rôle vérifié en base, jamais côté client seul) · journal des actions sensibles (prix, statuts, remboursements) · UI sobre réutilisant les primitives existantes, tableaux denses avec recherche/tri · aucune donnée de paiement manipulée (Stripe dashboard reste la référence).

## 5. Erreurs fréquentes à éviter

1. Publier un import sans réécriture : contenu dupliqué (SEO) et rupture du positionnement curation.
2. Vérifier les rôles côté client uniquement.
3. CMS externe pour 4 rôles internes : complexité sans bénéfice à cette échelle.
4. Oublier la revalidation ISR après édition (produit à jour < 60 s).

## Hypothèses

- **H41** : import par fichiers téléchargés (drag & drop `.html`/`.mhtml`) analysés hors ligne ; parseur défensif avec fallback saisie manuelle.
- **H42** : champ `role` sur l'utilisateur, comptes admin créés en base (pas d'auto-inscription admin).
