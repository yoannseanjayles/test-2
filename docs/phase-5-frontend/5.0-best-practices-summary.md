# Phase 5.0 — Best Practices Summary : Front-end

> **Statut** : 🟡 En attente de validation HITL
> **Équipe** : Architecte Front-end, UX/UI Designer, Expert SEO, QA
> **S'appuie sur** : specs Phase 2 (~60 composants), Design Guidelines 4.1 (D-045/D-047), objectifs D-009 (CWV 100 % Good, WCAG 2.2 AA, SEO 1ᵉʳ canal).

## 1. Choix de stack (proposition à valider)

| Couche | Choix | Justification |
|---|---|---|
| **Framework** | **Next.js 15 (App Router, React Server Components)** | Standard e-commerce 2026 : RSC = −40–60 % de JS client (INP/LCP), streaming SSR, `next/image` (LCP/CLS), metadata API (SEO). 0,1 s de vitesse gagnée ≈ +8 % de conversion (Google/Deloitte). |
| **Rendu** | **SSG + ISR** pour catalogue/éditorial ; SSR pour recherche ; client léger pour panier/checkout/compte | ISR : revalidation en secondes (vs rebuild complet) quand un produit change ; meilleure efficacité CDN qu'un SSR systématique. |
| **Langage** | TypeScript strict | Maintenabilité/reprenabilité (objectif projet). |
| **Styles** | **Tailwind CSS v4 + tokens CSS custom properties** générés depuis un fichier de thème unique | Les tokens 4.1 vivent en `@theme` ; zéro divergence design/code ; purge automatique = CSS minimal. |
| **Composants** | Bibliothèque interne (`/components/ui` primitives + `/components/commerce` métier), documentée en **Storybook** | Registre Phase 2 ; doc dans le code (D-045). Pas de kit UI externe : la DA est spécifique, les primitives sont simples. |
| **État client** | React Query (données serveur : panier, stock) + Zustand (UI locale : drawer, filtres) | Léger, standard, testé ; pas de Redux (surdimensionné). |
| **Formulaires** | React Hook Form + Zod (schémas partagés avec l'API en Phase 6) | Validation à la sortie du champ (D-033), types bout en bout. |
| **Tests** | Vitest (unit) + Testing Library (composants) + Playwright (parcours critiques : achat, retour) + axe-core (a11y automatisée) | Le tunnel d'achat est couvert par des tests E2E dès le début. |

**Hypothèse H37** : le back-end (Phase 6) exposera une API REST/JSON ; le front consomme via une couche `lib/api` typée (contrats Zod partagés) — le front reste agnostique de l'implémentation serveur.

## 2. Architecture de code

```
src/
├── app/                    # App Router : routes = sitemap 1.2
│   ├── (boutique)/         # layout boutique : header/footer
│   │   ├── page.tsx        # Accueil
│   │   ├── [animal]/       # /chien, /chat, /nac (+ [sousCategorie]/[produit])
│   │   ├── nouveautes/  recherche/  guides/  notre-histoire/
│   │   └── livraison-retours/  faq/  contact/  (+ légal)
│   ├── (tunnel)/           # layout épuré : panier, checkout (D-032)
│   ├── compte/             # layout compte (auth requise)
│   └── api/                # routes proxy si nécessaire
├── components/
│   ├── ui/                 # primitives : Button, FormField, Accordion, Badge…
│   └── commerce/           # ProductCard, MediaGallery, CartDrawer, FilterSidebar…
├── lib/                    # api client, utils, hooks, analytics
├── styles/                 # tokens (thème), ProseStyles
└── content/                # contenus éditoriaux (MDX) en attendant le CMS Phase 6/7
```

- **Server Components par défaut** ; `"use client"` uniquement pour l'interactif (drawer, facettes, formulaires) — la frontière est documentée par composant.
- Colocalisation : chaque composant = dossier (`Component.tsx`, `Component.stories.tsx`, `Component.test.tsx`).

## 3. Performance (CWV 100 % Good — D-009)

- **LCP < 2,5 s** : hero en `next/image` avec `priority` + `fetchpriority=high` ; AVIF/WebP automatique ; polices self-hostées (3 variables, D-046) préchargées avec `font-display: swap` + `size-adjust` sur les fallbacks.
- **CLS ≈ 0** : `aspect-ratio` réservé sur toute image (règle 4.1 §7), squelettes dimensionnés (D-027), zéro bannière injectée au-dessus du contenu.
- **INP < 200 ms** : RSC (peu de JS), imports dynamiques pour drawer/overlays, aucune lib > 30 KB sans justification consignée.
- Budget performance en CI : Lighthouse CI sur PR (seuils bloquants), bundle-analyzer surveillé.

## 4. SEO technique

- Metadata API par route (titles/descriptions des specs 2.1) ; `generateMetadata` dynamique produits/catégories.
- **Données structurées** centralisées (`lib/jsonld.ts`) : Organization, WebSite+SearchAction, Product, BreadcrumbList, ItemList, Article, FAQPage — conformes aux specs Phase 2.
- Sitemap XML segmenté généré (produits/catégories/éditorial, D-013), robots.txt, canoniques (règles D-026/D-028 : variantes sans URL, facettes en query-string canonisées).
- ISR + webhooks de revalidation : un produit modifié en admin est à jour en < 60 s sans rebuild.

## 5. Accessibilité (WCAG 2.2 AA)

- Socle : HTML sémantique (landmarks, hiérarchie de titres), primitives accessibles par construction (focus-trap overlays, `aria-live` panier/erreurs — specs 2.1), skip-link, `prefers-reduced-motion` global (4.1 §8).
- **axe-core dans les tests de composants** (violation = test rouge) + audit clavier manuel par page dans la checklist de dev.

## 6. Responsive

Mobile-first strict (D-003) : les composants sont développés mobile d'abord, breakpoints 4.1 (`sm/md/lg/xl`), barre de navigation basse mobile, méga-menu ≥ `lg` uniquement, zones tactiles ≥ 44 px vérifiées en revue.

## 7. Ordre de développement proposé (5.1)

1. **Socle** : projet Next.js + tokens 4.1 + polices + primitives UI (Button, FormField, Badge, Accordion…) + Storybook.
2. **Layout boutique** : Header/MegaMenu/MobileNav, Footer, AnnouncementBar.
3. **Pages par impact (D-019)** : Accueil → Fiche produit → Listing → Panier (drawer+page) → Checkout → Compte → Guides → Confiance/légal.
4. Données : mock API typée sur le catalogue démo (H33) — remplacée par l'API réelle en Phase 6 sans toucher aux composants (H37).

Chaque jalon = commit + démo ; HITL par lot de pages (proposition : 4 points de validation — socle+layout, cœur catalogue, tunnel, compte+contenus).

## 8. Erreurs fréquentes à éviter

1. Tout client-side (SPA) : LCP/SEO dégradés — RSC par défaut.
2. Kit UI externe re-thémé qui finit par imposer sa DA.
3. Tokens dupliqués entre Figma/CSS/JS — une seule source (`@theme`).
4. Tests E2E écrits « à la fin » — le tunnel est testé dès sa construction.
5. Hydratation massive par confort (`"use client"` en haut d'arbre).
