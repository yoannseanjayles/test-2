# Phase 6.0 — Best Practices Summary : Back-end

> **Statut** : 🟡 En attente de validation HITL
> **Équipe** : Architecte Back-end, Architecte Front-end, Expert Paiement, DPO/Juriste, QA
> **S'appuie sur** : contrats front H37 (types + schémas Zod de la Phase 5), modèle de commande D-016, hypothèses à confirmer (H10 3PL, H20 PSP, H21 livraison), objectifs D-009.

## 1. Choix de stack (proposition à valider)

| Couche | Choix | Justification |
|---|---|---|
| **Architecture** | **Monolithe Next.js** : Route Handlers + Server Actions dans le repo existant | Une seule base de code, déploiement Vercel inchangé, contrats Zod déjà partagés (H37) ; un backend séparé (Medusa, NestJS) est surdimensionné pour 1 boutique mono-devise — écarté, réévaluable si multi-canal |
| **Base de données** | **PostgreSQL managé (Neon)** + **Drizzle ORM** | Postgres = standard e-commerce (transactions, contraintes) ; Neon serverless s'accorde à Vercel (branches de préview) ; Drizzle : SQL typé léger, migrations versionnées |
| **Authentification** | **Better Auth** (e-mail + mot de passe, sessions) | Simple, auto-hébergé dans le monolithe (pas de SaaS d'auth), compatible checkout invité → rattachement de commande post-achat (D-014) ; pas de connexion sociale (H23) |
| **Paiement** | **Stripe** (Payment Element en iframe) + PayPal (H20) | Conformité PCI par iframe (D-033), Apple Pay/Google Pay inclus via Stripe, webhooks pour les statuts (D-016) ; prix recalculés serveur avant création du PaymentIntent |
| **E-mails transactionnels** | **Resend** + templates React Email | Confirmation/expédition/retour (D-016), restock (H15), relance panier (H9) ; visuels M-EML |
| **Contenus éditoriaux** | Guides en base (admin Phase 7) ; migration douce depuis `lib/guides` | Le gabarit E-E-A-T existe ; l'admin 7 éditera piliers/satellites (D-037) |
| **Recherche produits** | Postgres full-text (`tsvector` français) | Suffisant pour < 1 000 produits ; Meilisearch en évolution si besoin |
| **Validation & contrats** | **Schémas Zod partagés** front/API (déjà en place pour le checkout) | Types bout en bout ; H37 tenue : les composants front ne changent pas, seule `lib/api` bascule du mock vers la base |
| **Tests** | Vitest (unit + intégration API sur base éphémère Neon branch) + Playwright (tunnel avec Stripe test) | Le tunnel réel est couvert avant la mise en production |

**Hypothèses nouvelles** — **H38** : Neon (Postgres serverless) comme hébergeur BDD, réévaluable ; **H39** : stock décrémenté à la confirmation de paiement (webhook), réservation panier en évolution ; **H40** : relance panier (H9) = 1 e-mail à +4 h, opt-out simple.

## 2. Modèle de données (entités principales)

`products` / `product_variants` (taille × couleur, stock, prix — 2 axes max D-026/H14) · `categories` (taxonomie D-001/D-012) · `customers` (+ invités par e-mail) · `pets` (profil animal D-015, max 5 H24) · `carts` (invité 30 j, fusion à la connexion D-030) · `orders` / `order_lines` / `order_events` (statuts D-016) · `returns` (motifs, premier offert D-040) · `reviews` (achat vérifié D-025) · `restock_alerts` (H15) · `guides` / `authors` (D-037) · `settings` (livraison/seuils — config unique D-039) · `newsletter_subscribers` (consentement RGPD).

## 3. API & flux critiques

- **Catalogue** : lectures en RSC directes (Drizzle) + ISR revalidé par webhook admin (< 60 s, 5.0 §4).
- **Panier → commande** : Server Actions ; le total est **toujours recalculé serveur** (D-033) ; création PaymentIntent Stripe → webhook `payment_succeeded` → commande `Payée`, décrément stock (H39), e-mail confirmation.
- **Retours** : self-service ≤ 3 écrans (D-035) → étiquette (intégration 3PL, H10 à confirmer) → remboursement Stripe.
- **RGPD** : export/suppression self-service (D-035), consentements horodatés (D-041), durées de conservation (5 ans commandes).
- **Sécurité** : rate-limiting sur auth/contact, secrets en variables d'environnement Vercel, aucune donnée carte côté serveur (iframe PSP), audit des webhooks par signature.

## 4. Plan de développement 6.1 (jalons HITL)

1. **Socle données** : schéma Drizzle + migrations + seed du catalogue démo (H33) ; bascule de `lib/api` mock → base (le front ne change pas, H37).
2. **Auth + compte réel** : Better Auth, rattachement commandes invité, Mes animaux persistés.
3. **Paiement + commandes** : Stripe/PayPal, webhooks, e-mails transactionnels, suivi et retours réels.
4. **Contenus + finitions** : guides en base, restock/relance (H15/H40), recherche, RGPD, tests d'intégration complets.

## 5. Erreurs fréquentes à éviter

1. Faire confiance aux prix du client — recalcul serveur systématique (D-033).
2. Statuts de commande mis à jour hors webhooks (double source de vérité).
3. ORM verbeux + N+1 sur les listings — requêtes explicites, index sur les facettes.
4. Auth SaaS propriétaire difficile à quitter — contraire à l'objectif de reprenabilité.
5. Stock décrémenté à l'ajout panier (survente inverse) — au paiement confirmé (H39).
