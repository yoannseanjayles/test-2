# Phase 4.0 — Best Practices Summary : Design System

> **Statut** : 🟡 En attente de validation HITL
> **Équipe** : Directeur Artistique, UX/UI Designer, Motion Designer, Architecte Front-end
> **S'appuie sur** : registre de ~60 composants (Phase 2), palette DA verrouillée (D-044), objectifs WCAG 2.2 AA + CWV (D-009).

## Rôle dans notre contexte

Le Design System est le **contrat entre le design et le code** : il transforme les specs de la Phase 2 et la DA de la Phase 3 en un langage unique, réutilisable et documenté. Pour un projet destiné à être repris par n'importe quelle équipe (objectif projet), c'est le livrable le plus structurant.

## 1. Design tokens — l'architecture en 3 couches

Standard 2026 (Spectrum, Polaris, Material) :

1. **Primitives** : valeurs brutes nommées neutrement — `cream-100`, `caramel-500`, `space-4`, `font-size-300`.
2. **Sémantiques** : rôles — `color-bg-page`, `color-text-primary`, `color-action`, `color-feedback-error`.
3. **Composants** : `button-primary-bg`, `card-radius` — pointent vers les sémantiques.

Bénéfices : un changement de valeur se propage proprement ; le dark mode ou un rebranding = une nouvelle table de correspondance, pas une refonte. **Les contrastes sont validés au niveau des tokens sémantiques** (paires fond/texte testées AA une fois pour toutes).

Format : tokens en **CSS custom properties** générées depuis une source unique (JSON ou fichier de thème Tailwind) — compatible avec n'importe quelle stack (décision Phase 5).

## 2. Typographie

- **Rôles tokenisés** plutôt que tailles ad hoc : `display`, `heading-1..3`, `body`, `body-small`, `caption`, `label`, `price`.
- **Échelle modulaire** (ratio ~1.25 desktop, resserré mobile) avec tailles fluides (`clamp()`) — évite les breakpoints typographiques manuels.
- Combinaison premium classique : **serif expressive pour les titres** (personnalité, chaleur) + **sans-serif lisible pour le corps** (UI, prix, formulaires). Chargement : 2 familles max, `font-display: swap`, sous-ensembles latin, variable fonts si disponibles (perf D-009).
- Corps ≥ 16 px, interligne ≥ 1.5 pour la lecture longue (ProseStyles, spec Guides).

## 3. Couleur & accessibilité

- Point de départ : palette DA validée (D-044) — crème, greige, caramel, sauge, terracotta — étendue en gammes fonctionnelles (50→900) + couleurs de feedback (succès, erreur, alerte, info) accordées à la palette.
- **WCAG 2.2 AA systématique** : texte 4.5:1, texte large 3:1, composants UI 3:1 ; paires validées dans la doc des tokens. 94,8 % des homepages ont des violations détectables — l'exigence est différenciante.
- Ne jamais porter l'information par la couleur seule (états stock, erreurs : icône + texte).
- La couleur d'action (CTA) doit se détacher de la palette chaude : à définir en 4.1 (terracotta foncé ou vert profond — à trancher).

## 4. Grille, espacements, responsive

- **Échelle d'espacement en base 4 px** (`space-1`=4 → `space-16`=64) — tout espacement est un token, zéro valeur magique.
- Grille : 12 colonnes desktop (max-width ~1280–1440 px), 4 colonnes mobile, gouttières tokenisées.
- **Breakpoints** : `sm 640` / `md 768` / `lg 1024` / `xl 1280` — mobile-first (D-003), les layouts des specs Phase 2 s'y raccrochent (2 colonnes PDP ≥ lg, etc.).
- Zones tactiles ≥ 44×44 px (WCAG 2.2 target size + confort mobile).

## 5. Composants

- **Chaque composant du registre documenté avec** : anatomie, variantes, états complets (default, hover, focus, active, disabled, loading, error), comportement responsive, notes d'accessibilité (rôle ARIA, clavier), et *quand l'utiliser / ne pas l'utiliser*.
- **Le focus visible est une variante standard**, pas un ajout : anneau de focus tokenisé (`focus-ring`), jamais supprimé sans remplacement.
- Hiérarchie atomique pragmatique : primitives (Button, Input, Badge…) → composés (ProductCard, FormField…) → sections (Hero, FilterSidebar…) — alignée sur le registre Phase 2, pas de sur-ingénierie taxonomique.

## 6. Motion

- **Tokens de mouvement** : durées (`fast 150ms`, `base 250ms`, `slow 400ms`) et courbes (`ease-out` entrées, `ease-in` sorties, `ease-in-out` déplacements) — cohérence sans réinventer par composant.
- Doctrine premium : le mouvement **confirme et oriente** (feedback d'ajout panier, transitions de drawer), il ne décore jamais gratuitement. Ampleur faible (8–16 px), une seule propriété animée à la fois de préférence (transform/opacity — perf).
- **`prefers-reduced-motion` obligatoire** sur toute animation (déjà spécifié pages par pages en Phase 2 — devient une règle système).

## 7. Documentation & gouvernance

- La doc vit **dans le code** (Storybook ou équivalent — décision Phase 5) : usage, do/don't, a11y, exemples visuels.
- Toute divergence composant/spec = retour au Decision Log (pas de fork silencieux).
- Contribution : un nouveau composant n'entre au système que s'il est réutilisé ≥ 2 fois (règle anti-prolifération).

## 8. Erreurs fréquentes à éviter

1. Tokens à un seul niveau (`blue-500` utilisé partout) → rebranding impossible.
2. États manquants (pas de loading/error) découverts en développement.
3. Système documenté hors du code (Figma seul) qui dérive dès la 2ᵉ semaine.
4. Sur-animation « site vitrine » incompatible avec la performance et le premium sobre.
5. Accessibilité traitée en audit final plutôt qu'intégrée aux tokens/composants.
