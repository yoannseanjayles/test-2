# Phase 2.0 — Best Practices Summary : Checkout

> **Statut** : 🟡 En attente de validation HITL
> **Équipe** : UX Researcher, UX/UI Designer, Architecte Front-end, QA
> **S'appuie sur** : recherche 1.0 (fondamentaux checkout : invité, coûts, express — D-004, D-014), recherche Baymard dédiée au flow.

## Rôle dans notre contexte

Dernière ligne droite : tout ce qui précède (curation, confiance, panier) se convertit — ou s'évapore — ici. Objectif D-009 : abandon ≤ 55 % (vs ~70 % marché).

## 1. Structure du flow

- **Multi-étapes court et linéaire** : la recherche montre que le *nombre de champs perçu* compte plus que le nombre réel — un formulaire découpé en étapes logiques surperforme un formulaire monopage de +11–14 % en complétion. Notre panier moyen (~70–90 €) et la simplicité des produits permettraient le monopage, mais le multi-étapes gagne dès que la réassurance compte — cohérent premium.
- **Flow retenu à investiguer en spec** : `E-mail → Livraison → Paiement` (3 étapes), avec **récapitulatif visible en permanence** (colonne droite desktop, accordéon mobile).
- **Strictement linéaire** (Baymard) : jamais renvoyer l'utilisateur en arrière pour corriger ; l'édition d'une étape validée se fait en place, puis retour à l'étape courante.
- **Checkout épuré** : header simplifié (logo + réassurance, pas de navigation) pour limiter les sorties, mais logo cliquable (jamais piéger l'utilisateur).

## 2. Formulaires (le cœur du sujet)

- **8 champs suffisent** (la moyenne du marché est à 15, soit 2× trop) : e-mail, prénom, nom, adresse (autocomplete), complément (replié), code postal, ville, téléphone (optionnel, justifié : `pour la livraison`).
- **Autocomplétion d'adresse** (API type BAN/Google) + compatibilité totale avec l'autofill navigateur (attributs `autocomplete` corrects) ; code postal → ville pré-remplie mais éditable (28 % des sites mobiles l'omettent).
- **Validation inline après la saisie du champ** (pas pendant la frappe) ; en cas d'échec de soumission : données préservées + autoscroll vers l'erreur.
- **Carte bancaire** : formatage automatique avec espaces, validation Luhn front, détection du réseau (Visa/MC) par le premier chiffre, champs `inputmode` adaptés mobile.
- 1 colonne sur mobile, labels au-dessus des champs, boutons pleine largeur.

## 3. Confiance & paiement

- **Express d'abord** : Apple Pay / Google Pay / PayPal proposés dès l'étape e-mail (`ou payer directement avec…`) — court-circuitent tout le formulaire (recherche 1.0 : +12–15 %).
- Badges sécurité + rappel `Retours 30 jours` **adjacents au bouton de paiement** (+18 % de complétion, 1.0).
- Bouton final explicite : `Payer 84,90 €` (montant inclus — pas de `Continuer` ambigu).
- E-mail demandé en premier → relance panier possible (H9) + compte proposé post-achat (D-014).
- 3-D Secure (SCA obligatoire UE) : signalé avant la redirection (`Votre banque peut demander une confirmation`) pour éviter l'effet « page inconnue ».

## 4. Erreurs fréquentes à éviter

1. Création de compte demandée avant le paiement (D-014 la proscrit).
2. Coûts modifiés au checkout par rapport au panier (D-004/D-030 : tout est déjà connu).
3. Champs non compatibles autofill, ou validation agressive pendant la frappe.
4. Perte des données saisies après une erreur de paiement — cause d'abandon définitif.
5. Code promo proéminent (D-030 : replié).
6. Distractions : cross-sell, bannières ou navigation complète dans le tunnel.

## 5. Confirmation (fin du flow, User Flows 1.3)

- Récap complet (articles, adresse, délai estimé, contact), numéro de commande visible.
- **Création de compte en 1 clic** (mot de passe seul, e-mail connu) avec bénéfice explicite (`Suivez votre commande, gérez vos retours`) + onboarding profil animal différé (D-015).
- E-mail de confirmation immédiat (Phase 6).
